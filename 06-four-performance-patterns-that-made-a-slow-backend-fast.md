# Four Performance Patterns That Made a Slow Backend Fast

The previous post fixed one query. This one is about the patterns that fixed everything else. What's worth noting about all four of them: none was tied to the MCP architecture I abandoned, and none was tied to the LangChain planner that replaced it. They worked under both. That's usually a good sign that a pattern is actually a good idea, rather than just a good idea for one particular stack.

## Don't compute what you can precompute

The Dashboard's core design constraint was simple: it shouldn't scan the full 15-million-row table every time someone loads the page. So a batch of precomputed tables gets regenerated daily by an ETL pipeline, before anyone asks for anything:

| Table | Content |
|---|---|
| `daily_summary` | Daily totals, conflict/cooperation counts, sentiment, top actors and locations |
| `region_daily_stats` | Daily aggregates broken down by region |
| `geo_heatmap_grid` | Pre-aggregated geographic grid cells, ready for the map to render directly |
| `event_fingerprints` | Enhanced per-event records — title, summary, event type tags, severity score |

When the Dashboard loads, it checks these precomputed rows first. If they're there, it uses them directly for KPI cards and time series, which is how the platform hits its sub-200ms target. `event_fingerprints` deserves a specific mention: it's the bridge between a raw event record and something a person can actually read, and it comes in two tiers — a complete version generated by the ETL pipeline, and a lighter, real-time fallback generated on the spot when the precomputed version isn't available yet. The fallback isn't as rich, but it means the feature never just breaks.

## Four patterns that transfer across nearly any backend

Underneath the precomputed tables, four more general patterns did most of the remaining work. Nothing here is exotic. What made these effective was applying them consistently, everywhere the shape of the problem matched.

**Query caching, with a short TTL.** The same query run twice in a row shouldn't hit the database twice.

```python
rows = await query_cache.get_or_fetch(query=..., fetch_func=..., ttl=300)
```

A cache hit takes the query time from around 125ms down to about 0.05ms — a 2,500x difference — for a storage cost of roughly 2KB per cached entry, capped at 256 entries. In production, the hit rate settled around 87%. The TTL isn't uniform: static reference data gets cached for a full day, while real-time dashboard queries expire after a minute, since staleness matters more there.

**Parallel queries instead of sequential ones.** The Dashboard needs four or five independent pieces of data to render — KPIs, time series, top actors, a heatmap. Running them one after another took about 1,850ms. Running them concurrently with `asyncio.gather` brought that down to roughly 420ms, a 4.4x improvement, simply by acknowledging that none of these queries actually depend on each other.

**Streaming instead of loading everything into memory.** Calling `fetchall()` on a query that returns 100,000+ rows is a fast way to run out of memory. A server-side cursor reads the result in chunks instead:

```python
async for row in stream_query("SELECT ... LIMIT 100000", chunk_size=100):
    process(row)   # memory stays around ~100 rows
```

Memory usage dropped by about 90% — from roughly 500MB down to 50MB — and, more importantly, stopped scaling with result size at all. The dataset can grow without the memory footprint growing with it.

**Aggregation in the database, not in application code.** Pulling 100,000 rows into Python just to compute an average or a count is doing the database's job in the wrong place. Letting MySQL's `GROUP BY`, `AVG`, and `COUNT` do that work directly cut a roughly 850ms operation down to about 120ms — a 7x improvement — while also cutting data transfer by about 95%, since only the aggregated result crosses the wire instead of every underlying row.

## What all four add up to

| Optimization | Improvement |
|---|---|
| Serial → parallel queries | 4.4x |
| No cache → cache hit | ~2,500x |
| Python-side aggregation → DB-side aggregation | 7x |
| Single query → batched query | 11.6x |
| Direct LLM call → router pre-filtering | 3.3x (simple queries) |

That last row is worth pointing out, because it's the one pattern here that isn't purely a database technique — it's the same instinct applied to LLM calls. Filtering a request through a cheap local router before deciding whether it needs an expensive remote model is the same idea as caching or precomputing: do the expensive thing only when you actually have to.

All four patterns survived the MCP-to-LangChain pivot without needing any changes, because they're not really about MCP or LangChain at all — they're about not doing unnecessary work, and about doing necessary work in the place best equipped to do it. That's also why they were straightforward to apply consistently: with every query routed through the single SQL layer I described a couple of posts back, adding a cache or a streaming cursor to one query pattern meant the fix was available everywhere that pattern showed up, not just in the one place I happened to notice it.

At this point, the platform could serve fifteen million historical events fast, reliably, and without falling over under load. That solved the "what happened" half of the problem. It said nothing about the other half — what's likely to happen next — which is a fundamentally different kind of question. No amount of caching or indexing helps you forecast an event that hasn't occurred yet. That's where the project's focus shifted next: building a model that could actually look forward, instead of just serving up the past quickly. The next two posts are about that model — why I didn't just point an LLM at the problem, and how I got its prediction error down by half.
