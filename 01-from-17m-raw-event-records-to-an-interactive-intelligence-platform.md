# From 17M Raw Event Records to an Interactive Intelligence Platform

I started this project with about 17 million rows of data and a simple question: could I turn a dataset that almost nobody could query directly into something a person could actually talk to? By the end, I had a working answer, and one number made the case better than anything else could: on the hardest part of that answer — predicting what happens next — I cut the model's error in half.

That's the short version. Here's how I got there.

---

## The dataset nobody can just "look at"

The data source is GDELT — the Global Database of Events, Language, and Tone. It's a global feed that automatically extracts structured events from news coverage: who did what, to whom, where, and when. Each event comes with a CAMEO code describing the action, a Goldstein score rating how cooperative or conflictual it was, an average tone score capturing media sentiment, and a count of how many articles covered it.

It's rich data. It's also, in its raw form, close to unusable for a normal person asking a normal question.

Three problems stood out immediately:

1. **The fields are dense and the schema doesn't map to plain-language questions.** Answering something as simple as "what's the conflict trend in this region over the last month" means writing a long SQL query against fields most people have never heard of.
2. **Traditional dashboards only look backward.** They can summarize what already happened, but they can't tell you what's likely to happen next.
3. **Asking an LLM directly doesn't fix this.** Without real data behind it, a language model will answer confidently and get the specifics wrong. It has no grounding.

So the project goal became one sentence: build an interactive platform that combines a real database, an AI analysis layer, and a prediction model, so a user can explore events three different ways instead of one.

---

## Three pain points, three ways to look at the data

Each of the three problems above maps directly to a part of the platform:

- **Dashboard** answers "what happened" — KPI cards, daily event trends, top actors, and a geographic heatmap, all backed by precomputed aggregates so the page loads fast instead of scanning the full table on every request.
- **Forecast** answers "what's likely to happen next" — a 7-day risk forecast generated by a custom prediction model, not a dashboard trick. I'll get into how that model works, and how I got its error down by half, in a later post.
- **Analyst Chat** answers "let me ask a question in plain English" — natural language goes in, and the system routes it to either a precise database query or a narrative answer, depending on what's being asked. Every answer comes with a trace back to the data it's grounded in, so it's not just a confident-sounding guess.

None of these three pieces is complicated on its own. What makes the platform work is that they're deliberately kept separate, instead of trying to force one system to do all three jobs at once.

---

## What the final architecture looks like

The platform settled into four layers: data storage, a query service, an intelligence service, and the frontend.

```
Frontend (React + TS + Vite + ECharts + Leaflet)
   AI Explore  |  Forecast  |  Dashboard
        │           │            │
        ▼           ▼            ▼
FastAPI backend
   /api/v1/data/*     → DataService (fast path)
   /api/v1/analyze/*  → Planner / Executor (AI)
   /api/v1/data/forecast → Prediction service
        │                        │
        ▼                        ▼
MySQL 8.0                ChromaDB (semantic search)
(~17M events)             + prediction model checkpoint
```

One principle runs through the whole thing, and it shows up in nearly every technical decision I'll cover in this series: **keep fast, deterministic data access completely separate from LLM reasoning.** Dashboard queries never touch an LLM — they're structured lookups aimed at a sub-200ms response. Analyst Chat parses a natural language question into structured intent first, then either runs a precise query or, for open-ended questions, hands things off to a language model. Reports are generated lazily, only when a user actually asks for one. And every query in the system converges on a single shared query layer, so an optimization made once benefits every feature that touches the database.

---

## It didn't start out this way

Here's the part I want to be upfront about: this architecture is the result of a significant course correction, not the original plan. The project didn't begin with this planner-and-fast-path design. It began centered on a different protocol entirely, and partway through, I concluded that approach wasn't working and switched.

That decision — what the original design looked like, what specifically went wrong, and why I chose to change course mid-project rather than push through — is worth its own post. It's arguably the single most important decision in the whole project, and the next post covers it in detail.

For now, the platform runs: roughly 17 million events queryable in under a second, a forecast model with its error cut in half, and a chat interface that answers questions with real data behind it. What follows in this series is how each piece actually got built — including the parts that didn't work on the first try.
